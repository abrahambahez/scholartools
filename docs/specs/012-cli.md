# spec: 012-cli — CLI for scholartools public API

## findings

The current entry point (`scholartools._shell:main`) is an interactive Python shell that exposes all public functions via locals. It is a development convenience, not a production CLI.

Public API audit (31 functions across 9 groups from `scholartools/__init__.py`):
- **refs**: `add_reference`, `get_reference`, `update_reference`, `rename_reference`, `delete_reference`, `list_references`, `filter_references`
- **discover**: `discover_references`
- **fetch**: `fetch_reference`
- **extract**: `extract_from_file`
- **files**: `link_file`, `unlink_file`, `get_file`, `move_file`, `list_files`, `prefetch_blobs`
- **staging**: `stage_reference`, `list_staged`, `delete_staged`, `merge`
- **peers**: `peer_init`, `peer_register`, `peer_add_device`, `peer_revoke_device`, `peer_revoke`, `peer_register_self`
- **sync**: `push`, `pull`, `create_snapshot`, `list_conflicts`, `resolve_conflict`, `restore_reference`

Dependencies: only stdlib + existing deps (pydantic, httpx, pdfplumber, anthropic, cryptography). No click/typer installed.
Entry points in `pyproject.toml`: `scht` and `scholartools-test` both point to `_shell:main`.

## objective

Replace the interactive shell with a production CLI using stdlib `argparse`. Maps all 31 public functions to subcommands grouped by domain. JSON output by default (machine-readable); `--plain` flag for human-readable tables. The `scht` entry point becomes the primary CLI. All errors surface via exit code and stderr JSON — no unhandled exceptions.

Commands that take a single JSON/string argument accept it either as a positional arg or from stdin (pipe-friendly: `wpaste | scht refs add`). When the positional arg is omitted and stdin is not a TTY, the CLI reads from stdin.

Output envelope:
- Single result: `{"ok": bool, "data": {...} | null, "error": string | null}`
- List result: `{"ok": bool, "data": [rows], "page_info": {"page": int, "page_size": int}, "error": string | null}`

## acceptance criteria (EARS format)

- when `scht --help` is called, the system must display all command groups with short descriptions
- when `scht refs add <json>` is called with valid JSON, the system must call `add_reference()` and print JSON `{"ok": true, "data": ..., "error": null}` with exit 0
- when `scht refs add <invalid-json>` is called, the system must print `{"ok": false, "data": null, "error": "<parse-error>"}` with exit 1
- when `scht refs list` is called, the system must return paginated JSON with `page_info`
- when `scht refs list --plain` is called, the system must return a human-readable table with citekey, title, authors, year, ref_type columns
- when `scht refs filter --query Q --author A --year Y --type T --has-file --staging` is called, the system must pass all predicates to `filter_references()`
- when `scht discover <query> --sources crossref,arxiv --limit 5` is called, the system must call `discover_references()` with the parsed sources list and limit
- when `scht files link <citekey> <path>` is called, the system must call `link_file()` and return result JSON
- when `scht staging merge --omit k1,k2 --allow-semantic` is called, the system must pass `omit=["k1","k2"]` and `allow_semantic=True` to `merge()`
- when `scht sync push` is called and sync is not configured, the system must return `{"ok": false, "error": "sync not configured"}` with exit 1
- when `scht peers init <peer-id> <device-id>` is called, the system must call `peer_init()` and return result JSON
- when a command requiring a JSON arg is called without it and stdin is not a TTY, the system must read the argument from stdin
- when `echo '{"title":"X"}' | scht refs add` is called, the system must behave identically to `scht refs add '{"title":"X"}'`
- when a command requiring a JSON arg is called without it and stdin is a TTY, the system must exit with code 2 and print usage
- when any command fails (validation or runtime error), the system must exit with code 1 and include the error in the JSON `error` field
- when any command succeeds, the system must exit with code 0

## ADR required?

No. Stdlib argparse, no new dependencies, no architectural changes.

## risks

1. **JSON on the CLI.** Commands like `refs add` require JSON as a CLI arg. Mitigation: document quoting; defer `--from-file` / stdin support to follow-on.
2. **Table formatting edge cases.** Long titles/author lists will overflow fixed-width columns. Mitigation: truncate with `…` at a reasonable width (60 chars).
3. **Shell breakage.** Changing the `scht` entry point breaks existing shell users. Mitigation: keep `_shell.py` and `scholartools-test` entry point intact.
4. **Unhandled exceptions from public API.** Mitigation: bare `except Exception` at CLI dispatch level; log traceback to stderr in debug mode only.
