# product: loretools

## what a reference is

A reference is not a database record. It is a *claim structure* with four properties:

- **existence claim** — it asserts that something exists or existed: a text, a dataset, a recording, a textile, a performance, an interview, a field observation, an ephemeral event
- **retrieval promise** — it commits to some degree of traceability: archival (stable), URL at time of access (fragile), oral/documented (non-retrievable but witnessed)
- **authority type** — it carries a particular kind of epistemic warrant: peer-reviewed, community-validated, field-observed, legal, archival, artistic, journalistic
- **epistemological framework** — it is embedded in a knowledge system with its own standards of evidence: Western academic, indigenous, legal, activist, clinical

Michael Buckland's "Information as thing" is the operative principle: anything can be made into a reference if it is treated as a document — made retrievable, given a stake in a claim. Susan Star's boundary object analysis explains why this matters: existing reference infrastructure encodes one community's conventions (the Western research article with a DOI) as universal, and every other form of knowledge has to translate or be excluded.

loretools treats the reference as an epistemic object capable of contesting those asymmetries. CSL-JSON is a supported encoding, not the definition.

## what it is

loretools is a Python library for managing epistemic objects and the knowledge derived from them — built exclusively for AI agents. It is structured as a **portable core** extended by **optional plugins**.

The **core** is guaranteed to work identically everywhere Python runs. It has zero network dependencies, zero external authentication requirements, and deterministic behavior. It is useful in isolation — in air-gapped environments, in sandboxed agents, in contexts where network calls are blocked. The core is what you get when you install `loretools` with no extras.

**Plugins** extend the core for specific environments and use cases. A search plugin adds the ability to query Crossref or Semantic Scholar. A cloud plugin adds a DynamoDB/S3 backend. A sync plugin adds distributed peer coordination. An LLM plugin adds vision-based PDF extraction. None of these are privileged — a plugin for a community archive has exactly the same architectural standing as one for an academic index. Plugins are installed as needed, or not at all.

The library operates across two coupled layers:

- **Reference layer** — the epistemic objects themselves: CSL-JSON metadata, associated files, authority types, retrieval promises. This is the ground truth. Nothing in the knowledge layer is authoritative without a traceable reference here.
- **Knowledge layer** — markdown documents derived from reference content: synthesis notes, concept definitions, relation maps, reading paths. These are generated by agents reading reference files and are always linked back to their source objects.

References flow through two states: **staging** (exploration and evaluation) and **library** (production). Every record is mapped to a common internal model at intake — format ambiguity and source-specific conventions never propagate downstream. Promotion from staging to library is gated by a merge/QA step that validates schema completeness, detects duplicates, moves files to the archive, and assigns a citekey. Knowledge documents are built on top of committed library records — never staging.

## who it's for

The primary user is the agent-human dyad operating in any research-intensive context. The agent is the direct consumer of the library; the human is the beneficiary. Our optimization target is the dyad as a unit: agent effectiveness (reliability, token efficiency, composability) in service of human goals (zero search-management friction, improved discoverability, better curation, expanded thinking space). Secondary users are developers building research-assistant products who need a dependency they can trust.

## the core problem

Existing reference managers — Zotero, Mendeley, JabRef — are designed for human hands: GUIs, manual workflows, personal conventions. When an AI agent tries to assist with research, it has to work around tools that were never designed for it:

- No programmatic interface built for agent consumption — CLIs and APIs are afterthoughts
- Human workflows are informal and personal, making them unreliable for agents to replicate
- No standard for what "a reference operation" looks like as an atomic, composable unit
- Deduplication, auditing, and citekey management require manual human judgment today
- There is no local-first reference store an agent can own and operate without a running GUI application
- Agents cannot reliably extract references from PDFs or manage the file archive alongside the metadata
- No staging area exists for references under evaluation — everything is committed immediately, making exploration messy and libraries drift into inconsistency
- Format normalization happens late or not at all, so bib files and ad-hoc formats accumulate errors that compound over time
- Existing systems encode Western academic conventions as universal: a DOI-backed journal article is the implicit default shape of knowledge, with no first-class support for datasets, recordings, interviews, field observations, community archives, or ephemeral events — and no way to express the epistemological framework in which a source makes its authority claim

## what it does

**Core — reference layer** *(always available, no plugins required)*
1. extract — parse a local PDF or ebook and extract bibliographic metadata via pdfplumber, normalized to CSL-JSON
2. stage — add references to the staging area from any source: local file, manual entry, or data passed from a plugin; all records share a common internal model from intake
3. merge — promote staged records to the library after QA: schema validation, duplicate detection, file archival, citekey assignment
4. store — CRUD operations on the production library (`~/.loretools/library.json`)
5. files — manage associated files (`~/.loretools/files/`): link, move, rename, audit
6. citekeys — generate, assign, and manage human-readable citekeys consistently
7. deduplicate — detect and resolve duplicate records
8. audit — validate library integrity (schema issues, missing fields, orphaned files) and report to the agent
9. export — serialize references for external tools: pandoc-compatible CSL-JSON, Word XML (OOXML), Google Docs citation XML

**Core — knowledge layer** *(always available, no plugins required)*
10. read — extract and index full text content from archived reference files for agent consumption, using local extraction only
11. synthesize — generate synthesis notes across a set of references: summaries, arguments, tensions, open questions — stored as markdown linked to source citekeys
12. relate — map conceptual relationships between references and concepts; build a navigable relation graph stored as markdown
13. wiki — manage the knowledge wiki (`~/.loretools/wiki/`): create, update, link, and query concept pages derived from library content
14. annotate — attach structured annotations to individual reference objects; annotations are part of the reference record, distinct from wiki documents

**Plugin-provided operations** *(require explicit plugin installation)*
- **search** `[plugin: loretools-search]` — query external reference indexes (Crossref, Semantic Scholar, OpenAlex, community archives) and return staged candidates; the plugin defines which sources it speaks to — no academic index is built in by default
- **fetch** `[plugin: loretools-search]` — retrieve full bibliographic metadata for a known identifier (DOI, arXiv ID, ISBN) from external APIs
- **LLM extraction** `[plugin: loretools-llm]` — vision-based fallback extraction for scanned or complex PDFs where local extraction fails; also enables LLM-assisted synthesis and concept inference in the knowledge layer
- **cloud backends** `[plugin: loretools-cloud]` — DynamoDB/S3, MongoDB/GCS storage backends; switching from local to cloud requires this plugin
- **sync** `[plugin: loretools-sync]` — distributed peer coordination, blob sync, multi-device library state

**Interface**
All capabilities exposed as direct Python functions. Calling a plugin-provided operation without the plugin installed returns a `PluginNotAvailable` result — never a crash. Other distribution channels (MCP server, REST API) are secondary.

## storage

The local backend is built into core: `~/.loretools/library.json` for metadata, `~/.loretools/files/` for PDFs and EPUBs. It requires nothing beyond the filesystem.

Cloud backends are provided by the `loretools-cloud` plugin. Installing the plugin and setting the backend in `config.json` is the only change required — the function interface is identical. Core code never references cloud backends.

## non-goals

- No graphical or human-operated UI
- No free-form personal note-taking — the wiki is a structured knowledge layer derived from reference content, not a general scratchpad
- No knowledge objects without source references — every wiki page, synthesis note, and relation must trace back to library records
- No Zotero/Mendeley sync or migration tooling (could be a plugin; not a core concern)
- CLI is not a primary interface — it may exist as a thin wrapper but is not the design target
- No academic search built into core — if `loretools-search` is never installed, the library still has full value for working with references obtained through any other means

## success criteria

**Core only (no plugins required):**
1. receive a local PDF/ebook or manually-provided reference data and stage it — normalized to CSL-JSON immediately
2. run a merge/QA pass on staged records: validate schema, detect duplicates, archive files, assign citekeys — then promote to the library
3. retrieve, filter, and cite library references on demand with consistent citekeys
4. audit the library: report integrity issues, missing fields, orphaned files
5. export references for writing tools: pandoc CSL-JSON, Word XML, Google Docs XML
6. read the full content of archived reference files and produce synthesis notes and concept pages in the wiki, linked to source citekeys
7. query the wiki to surface concepts, tensions, and gaps across the collection

**With plugins:**
8. (search plugin) query external sources, stage results without committing, and present to the human for selection
9. (cloud plugin) switch storage backend via config without changing any agent code
10. (sync plugin) coordinate library state across devices or agents

The reference layer is the ground truth for committed epistemic objects. The knowledge layer is the agent-built understanding derived from them. Both are fully functional without any plugin installed.
